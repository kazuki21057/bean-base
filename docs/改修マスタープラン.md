## 1. 改修の全体方針

| # | 柱 | 概要 | 優先 |
|---|---|---|---|
| A | データ基盤変更 | Firestore → **Google Sheets** に戻す（管理を楽に） | 1 |
| B | 画面構成・遷移の再編 | 抽出と評価を分離、14画面構成へ再設計 | 2 |
| C | UI全面刷新 | ダッシュボード/抽出/評価/スタッツを一から。黒板風・瓶ビジュアル等 | 3 |
| D | その他軽微な修正 | レスポンシブ確認・DRY化・仕上げ | 4 |
| E | AI活用拡大 | PCAの説明範囲拡大など（**全改修完了後**） | 5（最後） |

**データ基盤の決定（A）:** 個人利用・データ量小のため、以前の `SheetsService`（`lib/services/sheets_service.dart`）を再活用して Google Sheets に戻す。スプレッドシートで直接データ確認・手動編集できる点が「管理が楽」の要件に最適。Cycle 18 で入った `FirestoreService` 経由の読み書きを Sheets 経由へ差し戻す。画像保存先は別途方針決定（後述 T0-4）。

**モバイル利用の決定(D、2026-07-13):** ネイティブアプリ(Android/iOS)は**作らない**。Web 版をホスティングし、スマホ(Android)のブラウザからアクセスして**ホーム画面にピン留め(PWA として standalone 起動)**して使う。ホスティング先は既存 Firebase プロジェクト(`beanbase-app-2016`)の **Firebase Hosting** を第一候補とする(無料枠で十分・`firebase.json` が既にある)。デプロイは `flutter build web --release` → `firebase deploy` の手動運用から始める。注意: Gemini API キーは `shared_preferences`(ブラウザごとの localStorage)保存のため、スマホ側でも設定画面から再入力が必要。

**公開URL(T3-11で確定、2026-07-18):** https://beanbase-app-2016.web.app — 認証なしで誰でもアクセス可能。再デプロイは `flutter build web` → `firebase deploy --only hosting`。

## 5. 日次ループ運用（要点）

- **終了条件**（いずれかで停止）: ①タスク完了 ②連続3回失敗 ③当日コスト>$12 ④当日ターン数>=30
- コスト・ターン数は `.claude/hooks/loop_guard.js` が transcript から算出し `.claude/loop_state.md` に出力。**この数値が真実**。
- 連続失敗は `.claude/loop_failures.txt`（`<日付> <回数>` 形式）を自分で更新（失敗+1／成功0リセット。日付が変わると自動的に 0 扱い）。
- 停止時は新規着手せず、`NEXT_SESSION.md`（引き継ぎ）と本書の進捗表を更新してから終了。
- 詳細は `CLAUDE.md`「日次改修ループ運用ルール」を参照。

### 5.1 自動ループの省コスト運用方針（$12/日）

> 変更履歴: 2026-07-03 にコスト上限 $0.5→$1.5、ターン上限 10→30 に引き上げ。2026-07-04 にコスト上限 $1.5→$12 に引き上げ。

コスト上限 $12/日 を守りながら自動化するため、**1日=最小1サブタスク**に限定し、**実装日と評価日を分離**する。役割をクラウドとローカルで分担:

| 担当 | 起動 | 役割 | やること | やらないこと |
|---|---|---|---|---|
| **クラウド schedule** | 毎日 05:00 JST（PC不要） | **実装日** | 最小1サブタスクのコード実装 → `flutter analyze` → `flutter test` → commit/push | `flutter run`（外部通信不可・高コスト） |
| **ローカル loop** | PC起動時 05:00 JST | **評価日** | `git pull` → `flutter run` → Playwright で疎通・見た目を評価 → 結果を `NEXT_SESSION.md` に記録 | 新規コード改修（NG項目は次の実装日へ回す） |

**省トークン徹底（コスト抑制の要）:**
- ファイルは全読みせず対象箇所のみ読む。大きな画面は「骨組み→部品→評価」と分割。
- Playwright の `snapshot` は最小限、スクショは要所のみ。
- $12 到達で `loop_guard` が停止 → その時点までを commit し引き継ぎ、翌日継続。

**競合回避:** クラウドは実装してpush、ローカルは pull して評価のみ（feature コードを commit しない）ので、同時刻でも git 競合は起きない。ローカルが評価するのは「直近 push 済みコミット」。

### コスト算出式（参考）
```
cost = Σ_model (
    input_tokens          × 単価_in
  + cache_creation_tokens × 単価_in × 1.25
  + cache_read_tokens     × 単価_in × 0.1
  + output_tokens         × 単価_out
) / 1e6        （当日分のみ）
```

|---|---|---|
| Fable 5 | 10.00 | 50.00 |
| Opus 4.8 | 5.00 | 25.00 |
| Sonnet 4.6 | 3.00 | 15.00 |
| Haiku 4.5 | 1.00 | 5.00 |
