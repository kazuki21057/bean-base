## 7. タスク細分化の最適化(マスタープラン §3 の置換案)

### 設計原則

1. **1タスク = 1日次ループで完結するサイズ**(目安: 変更ファイル 1〜4、新規画面 1〜3 枚)。30ターン/$1.5 の予算内で「実装→analyze→test→run→commit」まで到達できること。
2. **各タスクに「終了条件」を必ず持たせる**(Phase 単位ではなくタスク単位)。終了条件は検証コマンドで機械的に確認できる形で書く。
3. **共通部品を先に作るタスクを独立させる**。特にマスター系 6 種(豆/ドリッパー/フィルター/メソッド/グラインダー)の「一覧・詳細・新規」は構造が同型なので、**汎用テンプレート化 → 各マスターへ適用**の順に分解する(現行 T1-5〜T1-7 の「マスター種別ごと」分解より合計コストが下がり、「全タブへ漏れなく実装」の invariant も構造的に満たされる)。
4. **依存列を明記**し、日次ループが「今日やれるタスク」を機械的に選べるようにする。
5. サイズ表記: S(半日/ループ余裕)、M(1ループ標準)、L(1ループぎりぎり — 着手前に分割を検討)。
