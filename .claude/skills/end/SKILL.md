# end — 日次ループのセッション終了

BeanBase 2.0 の「日次改修ループ」(`CLAUDE.md` §日次改修ループ運用ルール) における、1ループの締めくくり。停止時の作法は必ずこの順序で行う。

## 手順

1. **`NEXT_SESSION.md` を更新する**
   - 新しい「当日やったこと」節を先頭付近(既存の直近セッション節の上)に追加する。何を実装したか、ユーザーからの指示内容、変更したファイル、検証結果(`flutter analyze`/`flutter test`/`flutter run`の結果、ブラウザ目視確認の有無)を具体的に記述する。既存の節のスタイル(日付・タスクIDの見出し、実施内容の箇条書き)に合わせる。
   - タスクが未完了/失敗で終わった場合は、引き継ぎ内容(何が問題で、次に何を試すべきか)を明記する。
   - 「次回の着手点」節を、今回完了したタスクを踏まえて更新する(次に着手すべき依存関係の空いたタスクを反映)。
2. **`docs/改修マスタープラン.md` の進捗表を更新する**
   - §3 のタスク表で完了したタスクを ⬜→✅ に変更する。
   - §4(画面インベントリ)や §2(全体進捗サマリ)など、完了したタスクに応じて更新が必要な表があれば合わせて更新する。
3. **新しい教訓があれば `rules/verification.md` の「教訓 (Lessons Learned)」節に追記する**
   - 既存の箇条書きの形式(**太字の見出し**: 内容の説明)に合わせる。すでに書かれている教訓と重複する場合は追記しない。
   - 「教訓」とは、今回の作業で判明した非自明な事実・ハマりどころ・回避策であり、単なる作業ログではない(作業ログは NEXT_SESSION.md 側)。
4. **`docs/` 配下の Cycle 連番が正しいことを確認する**
   - `docs/` と `docs/archive/` の既存 `cycle_<N>_*` フォルダの最大 N を確認し、今回新規に作成した(あるいはこれから作成する)ドキュメントフォルダの番号が重複・欠番なく連番になっているか確認する。ずれていれば修正する。
5. **commit / push する**
   - まだコミットしていない変更があれば、ユーザーに確認の上でコミットし、`git push` する。
   - 直前の `git status` で意図しないファイルが混ざっていないか確認してからステージする。

## 注意

- 手順は (a) NEXT_SESSION.md 更新 → (b) マスタープラン進捗表更新 → (c) commit/push の順を厳守する(`CLAUDE.md` の運用ルールに明記された順序)。
- コスト超過・失敗回数超過などの終了条件で停止する場合も、新規タスクには着手せずこの手順のみを実行する。
- push は共有リモートへの反映であり、実行前に一度ユーザーに確認するのが望ましい(明示的に「確認不要、自動でpushして」等の指示がある場合を除く)。
